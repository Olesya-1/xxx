Тестирование информационных систем

Лекция. Введение в тестирование. Чек-листы.

Что такое чек-лист?

Чек-лист (Checklist) — это инструмент, который представляет собой структурированный список проверок (тест-кейсов или пунктов), которые необходимо выполнить для подтверждения того, что определенная функция, модуль или система в целом работает корректно. Это не пошаговая инструкция (как тест-кейс), а список целей или тем для проверки. Тестировщик, используя чек-лист, сам решает, как именно проверить тот или иной пункт.

Зачем нужны чек-листы в тестировании?

1. Структурирование работы: Помогают не упустить ключевые моменты при тестировании, особенно в условиях сжатых сроков.

2. Экономия времени: На создание чек-листа уходит меньше времени, чем на написание подробных тест-кейсов.

3. Гибкость: Дают свободу действий тестировщику. Он может менять порядок проверок и методы их выполнения.

4. Фокус на качестве: Позволяют сосредоточиться на основных, наиболее важных или рискованных областях продукта.

5. Простота использования: Чек-листы понятны и легки в освоении для всех членов команды.

6. Идеальны для регрессионного тестирования: Когда нужно быстро проверить, что новые изменения не сломали существующий функционал.

Как создавать эффективные чек-листы?

1. Анализ требований: Изучите техническое задание (ТЗ), пользовательские истории, спецификации.

2. Определение областей тестирования: Разбейте систему на основные модули (например: "Главная страница", "Корзина", "Оплата", "Личный кабинет").

3. Составление пунктов: Для каждого модуля сформулируйте ключевые вопросы и цели проверки. Используйте технику тест-дизайна (эквивалентное разделение, анализ граничных значений) для полноты покрытия.

4. Определение приоритетов: Отметьте, какие проверки наиболее критичны для бизнеса.

5. Рецензирование: Попросите коллег (тестировщиков, аналитиков) проверить ваш чек-лист на полноту и понятность.

Вывод

Чек-листы — это мощный и гибкий инструмент в арсенале тестировщика, особенно в рамках МДК.05.03. Они идеально подходят для быстрого покрытия основных сценариев, регрессионного тестирования и работы в условиях, когда требования часто меняются. Умение создавать качественные и comprehensive (всеобъемлющие) чек-листы — ключевой навык для эффективного тестировщика.

Баг-репорт. Заголовок баг-репорта, шаги воспроизведения, приоритеты, окружения, логи, скриншоты и скринкаст.

Баг-репорт — это основной документ, который обеспечивает коммуникацию между тестировщиком и разработчиком. Его цель — четко, точно и однозначно описать проблему, чтобы разработчик мог легко воспроизвести и исправить ее.

1. Заголовок (Title/Summary)

2. Шаги воспроизведения (Steps to Reproduce)

3. Приоритет (Priority) и Серьезность (Severity)

  • Серьезность (Severity) — это объективная оценка влияния бага на работу системы. Определяется тестировщиком.

o S1-Blocker (Критический): Полный отказ системы, потеря данных, безопасность.

o S2-Major (Высокий): Ключевая функция не работает, но система в целом работает.

o S3-Normal (Средний): Функция работает не совсем так, как ожидалось, или есть неудобство.

o S4-Minor/Trivial (Низкий): Опечатка, неверный отступ, мелкая косметическая проблема.

  • Приоритет (Priority) — это субъективное указание, насколько срочно нужно исправлять баг.

o P1-High (Высокий): Исправить в ближайшем билде.

o P2-Medium (Средний): Исправить до релиза.

o P3-Low (Низкий): Можно исправить, когда будет время.

4. Окружение (Environment)

Что указать:

  • Операционная система: Windows 11 Pro 22H2, macOS Sonoma 14.4, Android 14, iOS 17.3.1
  • Браузер и его версия: Google Chrome 122.0.6261.112, Safari 17.3, Firefox 123.0
  • Разрешение экрана: 1920x1080, 1440x900, 375x812 (iPhone 12/13/14)
  • Устройство: ПК, ноутбук Dell XPS 13, iPhone 15 Pro, Samsung Galaxy S23 Ultra
  • Версия приложения: 2.1.0
  • Дополнительно: Аккаунт пользователя (если важно), язык системы, включенные флаги/фичи.

5. Логи (Logs)

Где взять:

  • Консоль браузера (F12 -> Console): Там могут быть ошибки JavaScript (JS Errors), 404 ошибки (не найден файл), ошибки CORS.
  • Сетевые запросы (F12 -> Network): Посмотреть, какой HTTP-запрос упал (статус 500, 400). Сохраните этот запрос в формате HAR (есть кнопка экспорта).
  • Логи сервера: Для мобильных приложений или сложных кейсов
  • Логи мобильного приложения (adb logcat для Android, Console.app для iOS).

6. Скриншоты (Screenshots) и Скринкаст (Screencast)

  • Скриншоты: Показывают статичное состояние бага.
  • Скринкаст (видеозапись экрана): Показывает последовательность действий и динамическое поведение бага.

Тест-кейсы. Статусы кейс-тестов и тест-сьюты.

1. Что такое Тест-кейс (Test Case)?

Тест-кейс — это минимальная единица тестирования, представляющая собой formalized описание действий, условий и входных данных, необходимых для проверки конкретной функции или характеристики тестируемого объекта, а также ожидаемого результата.

2. Структура тест-кейса (Атрибуты)

Стандартный тест-кейс содержит следующие поля

  • ID / Уникальный идентификатор
  • Название / Заголовок (Title)
  • Предусловия (Preconditions)
  • Шаги (Steps)
  • Ожидаемый результат (Expected Result)
  • Постусловия (Postconditions)
  • Приоритет (Priority)
  • Серьезность (Severity)

3. Статусы выполнения тест-кейса

В процессе тестирования тест-кейс может иметь следующие статусы:

  • Passed / PASS
  • Failed / FAIL
  • Blocked / BLOCKED
  • Skipped / SKIPPED
  • Not Run / NOT EXECUTED

4. Что такое Тест-сьют (Test Suite)?

Тест-сьют (Набор тестов) — это совокупность тест-кейсов, объединенных общей логикой или целью.

Цели создания тест-сьютов:

  • Организация: Группировка кейсов по функциональным модулям (например, "Тесты для корзины покупок", "Тесты для личного кабинета").
  • Выполнение по цели: Создание наборов для конкретного типа тестирования.
  • Автоматизация: Группировка кейсов для автоматического запуска (например, "API-тесты", "UI-тесты главной страницы").

5. Жизненный цикл тест-кейса

  • Создание (Draft): Аналитик/тестировщик создает кейс на основе требований, используя техники тест-дизайна.
  • Ревью (Review): Кейс проверяют коллеги (другой тестировщик, разработчик, аналитик) на предмет корректности, полноты и отсутствия ошибок.
  • Доработка (Rework): Автор вносит правки по комментариям ревьюера.
  • Утверждение (Approved): Кейс готов к использованию.
  • Выполнение (Execution): Тестировщик запускает кейс и присваивает ему один из статусов (Pass, Fail, etc.).
  • Обновление/Снятие с эксплуатации (Update/Retire): Если функционал изменился, кейс необходимо обновить. Если функционал удален — кейс архивируется.
