Аутентификация vs. Авторизация: Краткое различие

Аутентификация (Authentication - AuthN): Процесс проверки подлинности пользователя. Ответ на вопрос "Кто вы?". Это проверка логина/пароля, отпечатка пальца, Face ID и т.д.

Авторизация (Authorization - AuthZ): Процесс проверки прав пользователя на выполнение определенных действий. Ответ на вопрос "Что вам разрешено делать?". Это проверка ролей (админ, модератор, пользователь) и разрешений (может ли он читать, писать, удалять).

Часть 1: Аутентификация соединений с сервером.

Распространенные методы аутентификации сервер-сервер или клиент-сервер:

  • API Keys (Ключи API)
  • JWT (JSON Web Tokens)
  • OAuth 2.0 / OpenID Connect
  • Аутентификация на основе сессий (Cookies)

Часть 2: Авторизация логинов для подключения к базам данных

Ключевые принципы:

  • Принцип наименьших привилегий (Principle of Least Privilege)
  • Использование разных учетных записей для разных целей
  • Безопасное хранение учетных данных

2 Лекция: Требования безопасности к серверам баз данных. Классы защиты

Часть 1: Многоуровневая модель безопасности (Defense in Depth)

1. Уровень сети (Network Security)

  • Принцип: «Спрятать» СУБД от всего мира.
  • Меры:

o Изоляция: Размещение БД в отдельном, изолированном сегменте сети (DMZ, VLAN). Никаких прямых подключений из интернета.

o Firewall (Брандмауэр)

o Шифрование трафика: Использование SSL/TLS для соединений между приложением и БД. Защита от сниффинга (перехвата тракта).

Уровень доступа к серверу (Host Security)

  • Принцип: Укрепить ОС, на которой работает СУБД.
  • Меры:

o Минимизация поверхности атаки: Удаление всех ненужных пакетов, служб, отключение неиспользуемых портов.

o Своевременное обновление: Регулярный патчинг ОС и СУБД для устранения известных уязвимостей.

o Дисковая шифровка: Full-disk encryption (e.g., LUKS, BitLocker) для защиты данных на физическом носителе.

o Отдельный пользователь ОС: Запуск СУБД от имени непривилегированного пользователя (не root!).

3. Уровень доступа к СУБД (Database Access Control)

  • Принцип: Аутентификация и авторизация.
  • Меры:

o Сильные пароли: Политика сложности паролей, регулярная смена.

o Отказ от учетных записей по умолчанию: Переименование или блокировка root/sa/postgres.

o Принцип наименьших привилегий (Least Privilege): Пользователь/приложение получает ровно те права, которые нужны для работы, и не более.

o Разделение обязанностей (SoD): Администратор БД, администратор безопасности, оператор резервного копирования — должны быть разные люди/учетные записи.

4. Уровень данных (Data Security)

  • Принцип: Защита данных даже при компрометации вышележащих уровней.
  • Меры:

o Шифрование данных

o Маскирование данных (Data Masking): Для тестовых и сред разработки. Реальные данные заменяются на правдоподобные фиктивные.

o Хэширование: Для хранения паролей обязательно использовать современные алгоритмы хэширования с солью (bcrypt, scrypt, Argon2).

5. Уровень аудита и мониторинга (Auditing & Monitoring)

  • Принцип: Обнаружение и расследование инцидентов.
  • Меры:

o Включение ведения логов: Логирование всех попыток входа (успешных и неуспешных), попыток доступа к бд, изменений схемы БД.

o Centralized Logging: Отправка логов на выделенный сервер (лог-сервер) для защиты от модификации злоумышленником.

o SIEM-системы: Использование систем для анализа логов и выявления аномалий (например, 1000 попыток входа за минуту с одного IP).

o Регулярный аудит: Проверка настроек безопасности, прав пользователей.

Пользователи и права доступа в БД

Рассмотрим пример с тремя типами пользователей:

Администратор базы данных: (в жизни:) это IT сотрудник, который работает с базой данных. (в базе данных:) Этот пользователь обладает полными правами на управление базой данных. Он может создавать, удалять и редактировать таблицы, устанавливать права доступа для других пользователей, настраивать систему безопасности и выполнять другие административные задачи.

Диспетчер / личный менеджер: (в жизни:) этот сотрудник звонит вам по телефону, чтобы предложить новую услугу / отвечает на ваши обращения в банк, подскажет вам о ваших операциях или подключенных услугах. (в базе данных:) У этого пользователя есть доступ к информации о клиентах, чтобы помогать им с операциями и решать возникающие вопросы. Он может просматривать историю операций, баланс счета, персональные данные клиента, но не может изменять их.

Супервайзер: (в жизни:) это руководитель личных менеджеров, он следит за их работой и даёт возможность изменять данные клиентов по их обращениям. (в базе данных:) У этого пользователя есть более расширенные права, чем у диспетчера. Он может просматривать информацию о клиентах, а также изменять их данные, например, адрес или контактную информацию. Кроме того, супервайзер имеет доступ к информации о сотрудниках, но не может редактировать их данные.

1. Введение в ролевую безопасность

В современных СУБД (SQL Server, PostgreSQL, Oracle и др.) управление правами доступа осуществляется через роли. Роль — это именованный набор привилегий, который можно назначить пользователю или другой роли. Это упрощает администрирование: вместо назначения прав каждому пользователю отдельно, права назначаются ролям, а пользователи включаются в нужные роли.

Существует два основных уровня ролей:

1. Роли уровня сервера — управляют правами на уровне всего экземпляра СУБД.

2. Роли уровня базы данных — управляют правами в рамках конкретной базы данных.

2. Роли уровня сервера

Эти роли предназначены для администрирования самого сервера баз данных. Они существуют вне конкретных баз данных.

3. Роли уровня базы данных

Эти роли действуют в контексте конкретной базы данных и управляют правами на доступ к ее объектам: таблицам, представлениям, процедурам и т.д.

3.1. Фиксированные роли базы данных

Это предопределенные роли с уже настроенным набором привилегий. Их нельзя изменить.

3.2. Пользовательские роли базы данных

Администратор может создавать собственные роли для реализации сложной политики безопасности, соответствующей бизнес-процессам.

4. Иерархия прав доступа и принцип наименьших привилегий

Права могут конфликтовать. Общее правило:

  • Запрет (DENY) имеет наивысший приоритет. Он перекрывает любое разрешение (GRANT).
  • Разрешение (GRANT) дает право на действие.
  • Лишение права (REVOKE) отменяет ранее выданное разрешение или запрет.

Принцип наименьших привилегий: Пользователю должна быть предоставлена ровно та степень доступа, которая необходима для выполнения его рабочих задач, и не более. Это ключевой принцип информационной безопасности.
